## Hardware Migration Details

The key insight is that the Modbus protocol is identical — same function codes (0x05 write coil, 0x01 read coils, 0x0F write multiple coils), same addressing (0x0000-based, 0x00FF for all). The only transport change is serial → TCP socket. Both Waveshare modules default to "transparent mode" which is Modbus RTU frames over TCP (CRC still included).

### 30-CH Module (Levels)
- Coil addresses: 0x0000–0x001D (channels 1-30)
- All-relay address: 0x00FF
- Default IP: 192.168.1.254, Port: 502
- No digital inputs
- Function codes: 0x05, 0x01, 0x0F, 0x03, 0x06

### 8-CH Module C (Core)
- Coil addresses: 0x0000–0x0007 (channels 1-8)
- Digital input addresses: 0x0000–0x0007 via function code 0x02
- All-relay address: 0x00FF
- Default IP: 192.168.1.254, Port: 502 (will need different IP from 30-CH)
- Supports dry/wet contact digital inputs

### Architecture Impact
- modbus_relay.py needs TCP socket transport alongside serial
- HardwareManager needs two relay instances (relay_core, relay_levels)
- New digital input reader for door sensor (function code 0x02)
- Config needs IP/port per module instead of single serial port
- UI screens (relay, motor, sensor) need updates for dual-module

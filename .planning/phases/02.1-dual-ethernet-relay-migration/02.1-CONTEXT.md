# Phase 02.1: Dual Ethernet Relay Migration - Context

**Gathered:** 2026-03-23
**Status:** Ready for planning

<domain>
## Phase Boundary

Migrate from single RS485 Modbus relay + GPIO door sensor to dual Waveshare Ethernet PoE relay modules (30-CH for levels, 8-CH Module C for core + digital inputs). Update transport layer, HardwareManager routing, config structure, and UI screens. RS485 and GPIO kept as fallback options.

</domain>

<decisions>
## Implementation Decisions

### Transport & connection model
- Per-module transport choice: each module independently configured as TCP or RS485 in config
- Persistent TCP connections: open socket at startup, keep alive, reconnect if dropped
- RS485 serial kept as fallback transport option per module

### Module identity in UI
- One relay settings screen with two sections: "Core Module" (8-CH) and "Levels Module" (30-CH)
- Channel numbers as primary labels (CH1, CH2, etc.) with function name as secondary info
- All 30 channels shown for levels module, with mapped channels visually highlighted
- Connection status header per section: IP, port, and connection state (green/red dot)

### Failure & recovery behavior
- App launches immediately at startup, retries module connections in background; UI updates when connected
- Partial operation: if one module is down, machine shows degraded status but stays running for unaffected functions (though both modules needed for a sale)
- Module connection failures logged AND surfaced as persistent alert on maintenance/debug screen with last error time

### Digital input door sensor
- GPIO door sensing kept as fallback option alongside Modbus digital input (configurable)
- Fast polling: 100-200ms interval for Modbus digital input reads
- Sensor settings screen auto-detects active method from config, shows which is active (read-only)
- Single door sensor input only (DI1) — no generic framework for all 8 digital inputs

### Claude's Discretion
- Reconnect + retry strategy on connection drop (safety consideration: avoid double-firing coils)
- RS485 class architecture (unified transport abstraction vs separate implementations)
- Mid-transaction failure behavior when levels module drops (safety-first for vending context)
- Exact poll timing within the 100-200ms range

</decisions>

<specifics>
## Specific Ideas

- Waveshare modules use "transparent mode" — Modbus RTU frames over TCP (CRC still included), same function codes as RS485
- 30-CH module: coil addresses 0x0000–0x001D, all-relay 0x00FF, default 192.168.1.254:502
- 8-CH Module C: coil addresses 0x0000–0x0007, digital inputs via function code 0x02 (addresses 0x0000–0x0007), default 192.168.1.254:502
- Modules will need different IPs since they share the same default
- Both modules support function codes 0x05 (write single coil), 0x01 (read coils), 0x0F (write multiple coils), 0x03 (read holding registers), 0x06 (write single register)

</specifics>

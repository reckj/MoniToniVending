## Summary

This phase migrates the Modbus transport layer from RS485 serial to persistent TCP sockets, adds a second relay controller instance (30-CH levels module), adds Modbus FC02 digital input reading for the door sensor, and updates all callers (HardwareManager, config, UI screens). The Modbus protocol itself is unchanged — same frame format, same function codes, same CRC. Only the transport channel changes.

The existing `ModbusRelayController` class uses raw serial with a hand-rolled CRC. The new TCP variant reuses the exact same frame construction and CRC function, replacing `serial.Serial.write/read` with `asyncio.StreamWriter.write` / `asyncio.StreamReader.read`. Python 3.13 is confirmed available; asyncio stream API (`open_connection`, `StreamReader`, `StreamWriter`, `wait_for`, `timeout`) is fully present.

**Primary recommendation:** Implement a `ModbusTCPTransport` and `ModbusRS485Transport` (thin transport classes), extract a `ModbusFrameBuilder` utility, then build `EthernetRelayController` and `ModbusDigitalInputController` on top. The existing `ModbusRelayController` (RS485) remains for fallback; both share the same Modbus frame logic. This gives clean separation without a complex abstract base hierarchy.

### Pattern 1: Modbus-over-TCP Transport (persistent connection)

**What:** Persistent asyncio TCP connection. Write raw RTU frame (with CRC) to the socket; read response bytes. No Modbus TCP MBAP header — Waveshare transparent mode passes RTU bytes unchanged.

**When to use:** `transport_type: tcp` in module config.

**Key detail confirmed from hardware spec:** Waveshare "transparent mode" = Modbus RTU frames (including CRC) sent over TCP socket. This is NOT the Modbus TCP standard (which uses MBAP headers and strips CRC). Send the same 8-byte FC05 command that the RS485 code already builds.

```python
# Source: project knowledge + asyncio stdlib docs
async def connect(self) -> bool:
    try:
        self._reader, self._writer = await asyncio.wait_for(
            asyncio.open_connection(self.host, self.port),
            timeout=self.timeout
        )
        self.status = HardwareStatus.CONNECTED
        return True
    except (OSError, asyncio.TimeoutError) as e:
        self.last_error = str(e)
        self.status = HardwareStatus.ERROR
        return False

async def _send_frame(self, frame: bytes) -> bytes:
    """Send raw RTU frame and read response."""
    async with self._lock:
        try:
            self._writer.write(frame)
            await self._writer.drain()
            response = await asyncio.wait_for(
                self._reader.read(256),
                timeout=self.timeout
            )
            return response
        except (OSError, asyncio.TimeoutError) as e:
            self.last_error = str(e)
            self.status = HardwareStatus.ERROR
            self._writer = None
            self._reader = None
            return b''
```

### Pattern 2: Background Reconnect Loop (Claude's Discretion — recommended approach)

**What:** A background asyncio task that sleeps on a backoff schedule and retries `connect()` when status is ERROR or DISCONNECTED. The relay command methods check connection status and return False immediately if not connected (no queuing, no retry within the command).

**Safety rationale for vending context:** If a relay command fails mid-transaction, we do NOT retry automatically. A coil that partially fired (TCP write succeeded but no ACK received) could fire again on retry. The safe pattern is: command fails → return False → caller (purchase flow) cancels the transaction. The reconnect loop only restores the transport; it does not replay failed commands.

```python
# Recommended reconnect strategy
async def _reconnect_loop(self):
    backoff_seconds = [2, 5, 10, 30]  # escalating, cap at 30s
    attempt = 0
    while self._running:
        if self.status in (HardwareStatus.ERROR, HardwareStatus.DISCONNECTED):
            delay = backoff_seconds[min(attempt, len(backoff_seconds) - 1)]
            await asyncio.sleep(delay)
            success = await self.connect()
            if success:
                attempt = 0
                self.last_error = None
            else:
                attempt += 1
        else:
            await asyncio.sleep(5)  # connected: check every 5s
```

This loop is started in `connect()` as a background task and cancelled in `disconnect()`.

### Pattern 3: Modbus FC02 Digital Input Read

**What:** Send FC02 (Read Discrete Inputs) to the 8-CH module's digital input addresses to get door sensor state. DI1 corresponds to address 0x0000.

**Response format:** 6 bytes — `[slave_addr, 0x02, byte_count, data_byte, crc_lo, crc_hi]`. The `data_byte` is a bitmask: bit 0 = DI1, bit 1 = DI2, etc. For DI1 only: `di1_state = (data_byte >> 0) & 1`.

**Confirmed by test:** `bytes([0x01, 0x02, 0x00, 0x00, 0x00, 0x08])` + CRC reads 8 DIs starting at 0x0000.

```python
# Source: Modbus RTU spec + verified by code test
async def read_digital_input(self, di_index: int = 0) -> Optional[bool]:
    """Read single digital input. di_index 0 = DI1."""
    cmd = bytes([
        self.slave_address,
        0x02,           # FC02: Read Discrete Inputs
        0x00,           # Start address high
        di_index,       # Start address low (0 = DI1)
        0x00,           # Quantity high
        0x01            # Quantity low (read 1 input)
    ])
    crc = modbus_crc(cmd)
    frame = cmd + bytes([crc & 0xFF, crc >> 8])
    response = await self._send_frame(frame)
    if len(response) >= 4:
        return bool((response[3] >> 0) & 1)
    return None
```

### Pattern 4: Dual-Module HardwareManager Routing

**What:** `HardwareManager` exposes `relay_core` (8-CH) and `relay_levels` (30-CH) as separate attributes. The convenience methods `spin_motor` and `unlock_door`/`lock_door` are updated to route to the correct module.

**Key change:** `motor.relay_channel` now addresses `relay_core`. `door_lock.relay_channels` now address `relay_levels`. Both are 1-indexed against each module's channel count.

```python
# Updated HardwareManager routing
async def spin_motor(self, duration_ms: int) -> bool:
    if not self.relay_core:
        return False
    motor_channel = self.config.vending.motor.relay_channel
    await self.relay_core.set_relay(motor_channel, True)
    await asyncio.sleep(duration_ms / 1000.0)
    return await self.relay_core.set_relay(motor_channel, False)

async def unlock_door(self, level: int) -> bool:
    if not self.relay_levels:
        return False
    channel = self.config.vending.door_lock.relay_channels[level - 1]
    return await self.relay_levels.set_relay(channel, True)
```

### Pattern 6: 150ms Poll for Modbus DI Door Sensor

**What:** The existing sensor screen already polls at `Clock.schedule_interval(..., 0.2)` (200ms). For Modbus DI the poll is an async Modbus read — the 150ms target is within the 100-200ms range from the decision. Use 150ms (0.15) as the interval; it leaves adequate margin for a 1.0s Modbus timeout without stacking calls.

**Safety:** Use a guard flag `_poll_in_progress` to skip a tick if the previous read hasn't completed. This prevents TCP read queue buildup if the module is slow.

### Anti-Patterns to Avoid

- **Using pymodbus TCP client:** `pymodbus` (installed at 3.11.4) sends Modbus TCP frames with MBAP headers. Waveshare transparent mode expects raw RTU frames with CRC. Using pymodbus TCP would send the wrong framing. Use raw `asyncio.open_connection` + manual frame building instead.
- **Global reconnect lock across both modules:** Each module has its own lock. A stalled reconnect on the levels module must not block core module commands.
- **Retrying failed relay commands:** Return False on send failure; let the caller cancel the transaction. Do not buffer or replay coil writes — risk of double-firing is unacceptable in a vending context.
- **Blocking serial read in async context:** The existing RS485 code uses `run_in_executor` to avoid blocking the event loop. Keep this pattern for RS485 fallback.
- **Accessing `hardware.relay` after refactor:** All callers that used `hardware.relay` must be updated to `hardware.relay_core` or `hardware.relay_levels`. A grep is needed during implementation.

## Common Pitfalls

### Pitfall 1: pymodbus TCP vs Waveshare transparent mode
**What goes wrong:** Using `pymodbus.client.AsyncModbusTcpClient` sends Modbus TCP MBAP-framed requests (6-byte header, no CRC). Waveshare transparent mode expects raw RTU bytes (same as RS485, CRC included). The module will not respond.
**Why it happens:** pymodbus TCP is the standard Modbus-over-TCP implementation; Waveshare chose a simpler "pass-through" approach.
**How to avoid:** Use raw `asyncio.open_connection` and send the same byte sequences the RS485 code already builds.
**Warning signs:** Module sends no response to TCP writes; RS485 works fine with same commands.

### Pitfall 2: Both modules share the same default IP (192.168.1.254)
**What goes wrong:** If both modules boot with default IPs, they respond to each other's traffic or one is unreachable.
**Why it happens:** Factory default for both Waveshare modules is 192.168.1.254:502.
**How to avoid:** Configure unique IPs on each module before deployment. Document this as a one-time hardware setup step. The config must also have different IPs for core vs levels module.
**Warning signs:** Only one module responds; connection to second IP fails.

### Pitfall 3: Stale `hardware.relay` references in callers
**What goes wrong:** After renaming `HardwareManager.relay` to `relay_core`/`relay_levels`, any file that still references `hardware.relay` will get `AttributeError` or `None`.
**Why it happens:** `relay_screen.py`, `motor_screen.py`, `maintenance_screen.py`, and `purchase_flow.py` all reference `hardware.relay` directly.
**How to avoid:** Grep for `hardware.relay` and `self.hardware.relay` across all files before marking the plan complete. Update all callers.
**Warning signs:** `AttributeError: 'HardwareManager' object has no attribute 'relay'` at runtime.

### Pitfall 4: FC02 response timing — read before data arrives
**What goes wrong:** `asyncio.StreamReader.read(256)` returns immediately with 0 bytes if the module hasn't responded yet.
**Why it happens:** TCP is stream-based; `read(n)` returns whatever is available, not exactly n bytes.
**How to avoid:** Use `asyncio.wait_for(reader.readexactly(6), timeout=self.timeout)` for FC02 reads (response is always 6 bytes for 1 DI read). Use `readexactly(8)` for FC05 echo responses.
**Warning signs:** Response parsing fails silently; `len(response) < 4` in digital input read.

### Pitfall 5: asyncio lock per module, not global
**What goes wrong:** A single global lock means a slow reconnect on the levels module blocks the core module from firing motor relays during a sale.
**Why it happens:** Easy to accidentally share one `asyncio.Lock()`.
**How to avoid:** Each `EthernetRelayController` instance has its own `self._lock = asyncio.Lock()`. The `HardwareManager` does not hold a cross-module lock.

### Pitfall 6: Pydantic v2 config model changes
**What goes wrong:** Adding new fields to config models requires updating both the Pydantic model (config.py) and the YAML files (default.yaml, local.yaml). Missing a field causes a Pydantic validation error on startup.
**Why it happens:** Pydantic v2 raises `ValidationError` for missing required fields.
**How to avoid:** Give all new config fields `default` values in the Pydantic model. The deploy-specific values (IPs) go in `local.yaml`.

### Pitfall 7: Double-firing coils on reconnect
**What goes wrong:** If a FC05 write was in-flight when the TCP connection dropped, the module may have received and executed it. On reconnect, replaying the command fires the coil a second time.
**Why it happens:** TCP is not transactional; partial writes may succeed at the module level.
**How to avoid:** On reconnect, do NOT replay any pending commands. Return False from all in-progress commands. Callers must re-evaluate state and initiate fresh commands if appropriate. This is the primary safety constraint from Claude's Discretion.

### Modbus utilities (extract to modbus_utils.py)

```python
# Source: existing modbus_relay.py — extract unchanged
def modbus_crc(data: bytes) -> int:
    """Calculate Modbus CRC16."""
    crc = 0xFFFF
    for byte in data:
        crc ^= byte
        for _ in range(8):
            if crc & 0x0001:
                crc = (crc >> 1) ^ 0xA001
            else:
                crc >>= 1
    return crc

def build_write_coil(slave_addr: int, coil_addr: int, state: bool) -> bytes:
    """Build FC05 Write Single Coil frame."""
    cmd = bytes([slave_addr, 0x05, 0x00, coil_addr,
                 0xFF if state else 0x00, 0x00])
    crc = modbus_crc(cmd)
    return cmd + bytes([crc & 0xFF, crc >> 8])

def build_read_di(slave_addr: int, di_addr: int, count: int = 1) -> bytes:
    """Build FC02 Read Discrete Inputs frame."""
    cmd = bytes([slave_addr, 0x02, 0x00, di_addr, 0x00, count])
    crc = modbus_crc(cmd)
    return cmd + bytes([crc & 0xFF, crc >> 8])

def build_write_all_coils(slave_addr: int, state: bool) -> bytes:
    """Build FC05 with 0xFF special address for all relays."""
    cmd = bytes([slave_addr, 0x05, 0x00, 0xFF,
                 0xFF if state else 0x00, 0x00])
    crc = modbus_crc(cmd)
    return cmd + bytes([crc & 0xFF, crc >> 8])
```

## Open Questions

1. **Slave address for both modules**
   - What we know: Waveshare default is slave address 1. Both modules will likely remain at address 1 since they're on separate TCP connections (different IPs), so the address is just a convention.
   - What's unclear: Whether the Waveshare module enforces the slave address in the RTU frame when running over TCP transparent mode, or ignores it.
   - Recommendation: Use slave address 1 for both, send it in the frame anyway, add it as a config field for flexibility.

2. **FC05 echo response on TCP**
   - What we know: RS485 code checks `len(response) >= 8` but notes "even without response, command may have worked". RS485 often echoes the sent frame.
   - What's unclear: Whether Waveshare TCP transparent mode echoes the FC05 command or sends a proper FC05 response, and whether it does so immediately.
   - Recommendation: Use `readexactly(8)` with a timeout. If timeout, log but still update local state cache (same approach as existing RS485 code).

3. **Exact Modbus slave address sent in DI reads vs coil writes**
   - What we know: FC02 read goes to the 8-CH module; FC05 coil writes go to either module.
   - What's unclear: Nothing blocking — both modules are on separate TCP connections, so the slave address in the frame could be 1 for both.
   - Recommendation: Confirmed non-issue. Keep slave_address: 1 for both, configurable.
